# Implement features

追加実装や機能実装を行う。

## Workflow

### 1. Planing

- [ ] テストリストを作成しユーザーと合意をとる。
  - テストリストはコンテキストに保持しておく。必要に応じてファイルに書き出す。
  - テスト環境が整備されてない場合スキップ OK
  - テストを [テスト設計の考え方](#テスト設計の考え方) に乗っ取って設計する。

- [ ] インタフェース設計を行う。
  - テストを最初に書く目的の 1 つはインタフェース設計を行うことである。
  - [インタフェース設計の考え方](#インタフェース設計の考え方) を参照する。

- [ ] コードベース全体のアーキテクチャを理解した上でどこに何を追加するのか明確にする。
- [ ] 仕様や要件について不明点があればユーザーへ質問する

### 2. TDD

1. TDD の 1 サイクルを回す

- 1 サイクル = 1 テスト
- Red: テストリストの中から 1 つ選んでテストを実装する
- Green: 通す。シンプルな実装で。
- Refactor: リファクタする。[リファクタリングガイド](refactoring.md) を必要に応じて参照する。

2. コミット・テストリストの更新

- 1 サイクル回ったらユーザーに通知
- コミット前に通知する
- 1 サイクルを回す中で新たに気づいたテストはテストリストに追加する。
- 対応済みのテストはリストから削除する

## Reference

### テスト設計の考え方

**良いテスト**

- ユーザー/呼び出し元が気にする振る舞いをテストする
- パブリック API のみ使用
- 内部リファクタリングに耐えうる
- HOW ではなく WHAT を記述する
- テストごとに 1 つの論理的なアサーション

**悪いテスト**

- 内部依存するプライベートモジュールのモック
- プライベートメソッドをテストする
- 内部モジュールやメソッドの呼び出し回数や順序をアサートしている
  - 例: "checkout が paymentService.process を呼ぶ" のようなテストケース
- 振る舞いが変わっていないのにリファクタリングでテストが壊れる
- テスト名が WHAT ではなく HOW を記述している
- モジュールのインターフェースを介さず外部手段で検証している
  - 例:
    Bad: DB を副作用に持つメソッドのテストで検証で SQL を叩く。
    Good: DB 呼び出しを含んだ取得メソッドを用いて検証する。

**モックの考え方**

- システム境界でのみモックするべき
- examples:
  - 外部 API (決済、メールなど)
  - データベース (場合によってはテスト用 DB をモックせず用意して使う)
  - 時刻 / 乱数
  - ファイルシステム (場合によって)
- モックすべきでないもの
  - 自分で実装するクラス/モジュール
  - 処理に必要な内部クラスやモジュール
  - 自分でコントロールできるもの全般

### インタフェース設計の考え方

**外部依存を注入可能に保つ**

- 依存を内部に閉じ込めず、外から注入可能にする。
- 関数の引数や interface による DI を用いる

**副作用を避け、純粋にする**

- 副作用を起こすのではなく、結果を返す
- 例外を投げずに値として返す (場合による)

**小さく深いインタフェース (Deep Module)**

- メソッドを少なくする
- パラメータを少なくする
- 実装を深くする。
